# v3.0 产品原则

> 状态：v3.0 目标原则。本文定义产品取舍，不描述实现迁移。

## 原则一：配置可理解、可迁移、可验证

配置是用户长期维护的资产。系统应让用户能理解当前配置、修改配置、验证配置、导入导出配置，并能明确知道配置是否已经生效。

设计要求：

- 每个持久字段都属于明确的配置语义模型。
- 保存动作必须返回新的 `config_revision`。
- 导入导出服务于迁移、备份和分享，不作为隐藏状态通道。
- 配置格式和存储方式不影响 Product Core。

## 原则二：生效状态必须清楚

用户需要知道“我正在编辑什么”和“服务正在使用什么”。工作配置、草稿配置和运行时快照不能混在一起。

设计要求：

- 保存不等于生效。
- reload 成功才创建新的 `RuntimeSnapshot`。
- dirty 状态必须在 UI 和 API 中可见。
- 运行时预览、生成和生效配置导出都必须来自快照。

## 原则三：错误是工作台的一部分

错误不是 panic 文本，而是指导用户修复配置的产品能力。

设计要求：

- 诊断必须有稳定 `code`。
- 诊断必须标明发生阶段。
- 能定位到配置字段时必须带 locator。
- 级联问题必须保留 cause path。
- 格式相关问题必须明确 target format。

## 原则四：扩展要有入口

新增来源、协议、目标格式、渲染能力或 UI 提示时，开发者必须能快速判断改哪里。

设计要求：

- 来源、协议和目标格式能力集中在 `CapabilityRegistry`。
- 具体 parser、projector、renderer 只在 binding registry 中绑定。
- Build 保持格式无关。
- Target Projection 统一处理格式能力差异。
- Render 只序列化已经合法的 TargetView。

## 原则五：模块化单体优先

v3.0 仍是单二进制产品。架构打磨的目标是清晰边界和高演化效率，不是过早拆成多个服务。

设计要求：

- Product Core 不依赖外部 I/O。
- Application Services 只编排用户动作。
- Engines 保持纯转换。
- Adapters 承接 HTTP、文件、远程资源、认证和部署环境。
- 默认不引入数据库或外部队列。
